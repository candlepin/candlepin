# CLAUDE.md

This file provides guidance to Claude Code when working with code in this repository.

## What is Candlepin

Candlepin is a system registration and content access engine. It is a Java WAR application deployed on Tomcat that exposes a REST API.
RHEL systems ("consumers") register via subscription-manager, and Candlepin generates X.509 certificates for them,
used for mTLS authentication against Candlepin itself, and other services (such as the CDN and Pulp) that provide content access.

It has 2 modes of deployment: Standalone (deployed as part of upstream Foreman/Katello/Pulp/Candlepin setup,
or as the Red Hat Satellite product) and Hosted (deployed as a Red Hat managed service).

## Quick Reference Commands

Gradle wrapper (`./gradlew`) is the build tool. Assumes JDK 25 is installed.

```bash
# Build
./gradlew war                                               # Build the WAR file
./gradlew clean                                             # Clean build artifacts

# Test
./gradlew test                                              # All unit tests
./gradlew test --tests org.candlepin.controller.Cdn*        # Single test class/pattern
./gradlew spec                                              # All spec (integration) tests
./gradlew spec --tests org.candlepin.spec.StatusSpec*       # Single spec test

# Lint
./gradlew checkstyle                                        # Checkstyle on all code
```

## Running Locally (Dev Container)

Assumes `podman` & either of `docker-compose` or `podman-compose` are installed.
```bash
./bin/deployment/deploy-container -HMag         # redeploy Candlepin & Postgres in Hosted mode
./bin/deployment/deploy-container -Ma           # redeploy Candlepin & Postgres in Standalone mode
podman container rm -f candlepin postgres       # Remove containers
```
Access at `https://localhost:8443/candlepin`.

## Architecture

### Frameworks & Stack

- **DI**: Google Guice (not Spring). Configured in `CandlepinModule`, `CandlepinFilterModule`.
- **REST**: JAX-RS via RESTEasy. Resource classes in `org.candlepin.resource` implement generated API interfaces.
- **Persistence**: Hibernate/JPA with C3P0 connection pooling. PostgreSQL (default) or MariaDB.
- **Database Migrations**: Liquibase. Add changeset under `src/main/resources/db`. Include the changeset here: `src/main/resources/db/changelog/changelog-update.xml`
- **Messaging**: Embedded ActiveMQ Artemis. Config in `src/main/resources/broker.xml`. Abstracted via CPM (Candlepin Pub/Messaging) layer.
- **Async Job system**: uses Artemis messaging (for distributing jobs on a Candlepin cluster setup), and job status is persisted in the DB.
- **Job Scheduling**: For scheduled jobs: Quartz with JDBC persistence.
- **Crypto/PKI**: BouncyCastle for X.509 certificate generation, key pairs, PEM encoding.
- **Config**: SmallRye Config. Template at `config/candlepin/candlepin.conf.template`, runtime at `/etc/candlepin/candlepin.conf`.
- **API Spec**: Spec-first: Add API changes in `api/candlepin-api-spec.yaml`. openapi-generator gradle plugin generates interface stubs. We implement those in `*Resource.java` classes.

### Application Entry Point

`CandlepinContextListener` (extends Guice's `GuiceResteasyBootstrapServletContextListener`).
Initializes config, runs Liquibase migrations, creates the Guice injector, and starts subsystems (Artemis, Quartz, etc.).

### Key Package Patterns

- `org.candlepin.model` — JPA entities and **Curators** (the DAO pattern; e.g., `ConsumerCurator`, `PoolCurator`)
- `org.candlepin.resource` — REST endpoints implementing generated `*Api` interfaces
- `org.candlepin.dto.api.server.v1` — Generated DTOs from OpenAPI spec
- `org.candlepin.controller` — Business logic (e.g., `PoolManager`, `Entitler`)
- `org.candlepin.service` — Adapter interfaces (these are supposed to be implemented elsewhere in the Hosted Candlepin use case)
- `org.candlepin.pki` — Certificate and key generation infrastructure
- `org.candlepin.async` — Job framework
- `org.candlepin.audit` — Messaging/Event system (publishes to / receives from Artemis)
- `org.candlepin.policy` — (Legacy) Business rules (JavaScript-based via Rhino).

### Deterministic Code Generation

Tasks such as `openApiGenerate` and `msgfmt` that autogenerate code are automatically run before `compileJava`.
Do NOT run them manually. Do NOT modify autogenerated code under `build/`.

## Testing

- **Unit tests required** for all externally accessible methods. Must cover: valid input, null, empty, bad input, and edge-case contracts.
- **Spec tests required** for features/APIs. Cover full lifecycle, minimize mocks. Run against realistic application config.
- When validating incremental changes, run unit and spec tests scoped to the affected package or resource class.
- Unit tests: JUnit 5, Mockito, AssertJ. Source in `src/test/java/`.
- Spec tests: functional tests using an auto-generated OpenAPI client. They require a running candlepin instance. Source in `spec-tests/src/test/java/`.

## Git Commits

- Use past tense verbs

## Code Style

- Style enforced by `./gradlew checkstyle`. Config: `config/checkstyle/checkstyle.xml`. Zero warnings/errors tolerance.

### Formatting

- **Indentation:** 4 spaces, no tabs.
- **Column limit:** 128 characters. Except for `import` statements.
- **Braces:** Stroustrup. Always use braces for `if`/`else`/`for`/`do`/`while`, even single-statement bodies.
- **Empty blocks:** Must contain a comment explaining the intentional emptiness (e.g. `/* intentionally left empty */`).
- **One statement per line.**
- **Fluent chains:** 2 or fewer invocations may be on one line. 3+ must break before each `.`, indented +4.
- **Line wrapping:** Break after all operators. Continuation lines indented +4.
- **Numeric literals:** Use uppercase `L` suffix (e.g. `3000000000L`).
- **Text blocks:** Opening `"""` on a new line; closing `"""` on its own line at same indentation.

### Imports

- **No wildcard imports** (static or otherwise).
- **No static import for nested classes** (use normal imports).

#### Import Order
- Static imports first (one block)
- Then non-static imports grouped by TLD or TLD+SLD in this order: `org.candlepin,ch,com,io,liquibase,net,okhttp3,org,tools,java,jakarta,javax`, separated by blank lines.
- Alphabetical within each block.

### Naming

- **Classes:** `UpperCamelCase`. Nouns/noun phrases. Test classes end with `Test`.
- **Methods:** `lowerCamelCase`. Verbs/verb phrases. Underscores allowed in test method names.
- **Constants** (`static final`, deeply immutable): `UPPER_SNAKE_CASE`.
- **Non-constant fields, parameters, locals:** `lowerCamelCase`.
- **Packages:** All lowercase, no underscores.
- **No prefixes/suffixes** like `_name` or `name_`.
- **Acronyms:** Treat as words (`XmlHttpRequest`, not `XMLHTTPRequest`; `newCustomerId`, not `newCustomerID`).
- **No punny/joke names** that obscure intent.

### Programming Practices

- **`@Override`:** Always use when legal.
- **No `var` keyword.** Explicit types required.
- **`this.` required** for all instance member references (avoids shadowing).
- **Static non-constant members** must be qualified by class name (e.g. `MyClass.types`).
- **Static constants from imported classes** qualified by class name (e.g. `Messages.DEFAULT_MESSAGE`).
- **Own constants** (same class, `static final`) do not need qualification.
- **Caught exceptions:** Never silently ignored; add a comment if intentionally empty.
- **Access modifiers:** Classes default to package-private. Instance members default to `private`. Methods should be as narrow as possible. Package-private instance members require a `/* pkg-private */` comment.

### Annotations

- **Class/method/constructor annotations:** One per line, before the declaration.
- **Parameter annotations:** Inline before the parameter type.
- **Field annotations:** One per line, before the field.
- **Type-use annotations:** On the line before the annotated type.

### Javadoc

- Required for all `public`/`protected`/`package` members of visible classes. Optional for self-explanatory members. Not required on overrides unless behavior changes.
- **Summary fragment:** Noun/verb phrase, not a full sentence. No "This method returns..." phrasing.
- **Block tag order:** `@deprecated`, `@param`, `@throws`, `@return`. No empty descriptions.
- **`@param`/`@throws`:** Parameter on same line as tag, description on next line indented +4 from Javadoc indent.
- **Paragraphs:** Separated by a line containing only `<p>` (unclosed).
- **HTML indentation in Javadoc:** +2 per nesting level.

### Spec Test Practices

- Annotate test classes with `@SpecTest`.
- Tests must be isolated and safe to run in parallel, in any order.
- Avoid `beforeEach`/`beforeAll` for trivial data creation; create data locally in each test.
- Shared data (via `@BeforeAll`) is only acceptable when tests are read-only against it; encapsulate in
 a `@Nested` class with `Lifecycle.PER_CLASS`.
- Don't write cleanup routines unless truly global state is affected.
- Use `@OnlyInHosted` / `@OnlyInStandalone` for mode-specific tests; avoid runtime mode checks.
- Test method names: terse camelCase describing target and behavior. No `@DisplayName`.
- Don't rely on count-based assertions that break when parallel tests add data.
